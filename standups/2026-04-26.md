## Completed (PrivateCore — privatecore-ios `main`)
- **Home crash fix** — `BriefingView` triggered HealthKit `requestAuthorization` without `NSHealthShareUsageDescription` in Info.plist; iOS hard-killed the app on tab tap. Decided with Miguel to drop the Health section from PC-060 entirely (out of scope for PrivateCore's value prop). Removed the call site + Health UI.
- **Library categories** — Intelligence row (Trips/Places/Boards/Insights/People) was hidden behind `if filter.isEmpty` *inside* `cardListContent`, so empty libraries showed only the "tray" empty state with no category nav. Lifted Intelligence to always render at top of LibraryView.
- **CLIP autostart** — `PrivateCoreApp.swift` only kicked OCR + EmbeddingService on `scenePhase == .active`; CLIP was missing. Plus the `.onChange` handler doesn't fire on cold launch (no scenePhase transition). Replaced with an `init()`-spawned permanent watchdog Task that calls `kickPendingPipelines()` every 60s for the lifetime of the process.
- **Schema migration ordering** — `migrateSchema` did `ALTER TABLE boards` *before* `CREATE TABLE IF NOT EXISTS boards`, so any DB pre-dating the boards table threw on column-check and aborted the entire migration. Reordered: all `CREATE TABLE IF NOT EXISTS` run first, then column ALTERs.
- **`trips.cached_summary`** — PC-053 added the column to the model + SQL queries but never to the schema. Console flooded with `no such column: cached_summary` on every fetchTrips. Added explicit ALTER TABLE migration + included in CREATE TABLE.
- **CLIP text feature name** — `embedText` passed `"input_ids"` as the dict key but `mobileclip_s0_text.mlmodelc` expects `"text"`. Every CLIP visual search has been throwing `"Feature text is required but not specified"` and silently returning 0 results. One-character key fix.
- **Add Sample Cards** — `addSampleCards` was calling `try? SQLiteStore.shared.upsertCard(card)` without `await`. Since SQLiteStore is an actor, the call returned without executing. Added `await`.
- **New Board → Create refresh** — `upsertBoard` wasn't posting `SQLiteStore.didChangeNotification`, so LibraryView never refreshed after creation; UI looked broken. Added the post.
- **PeopleView regression** — peer commit `ddc0f5d` rewrote PeopleView for fault-tolerant load (PC-072) but introduced `\\(count)` literal-escape bug + dropped `NewPeopleGroupSheet` and `PeopleGroupCard` stubs. Fixed escape, restored stubs, fixed `insertBoardMember` arg label drift (`sourceType` → `cardType`).
- **Misty's escape errors** — `\/` is not a valid Swift escape sequence (it's JSON/JS), and `\\$\{...)` was a mangled interpolation. Both crashed the build with the build was attempted. Patched the canonical CardTitleSanitizer + the print statement in SQLiteStore.
- **ShareExtension build unblock** — added a fileprivate duplicate of `CardTitleSanitizer` in `ShareViewModel.swift` since `PrivateCore/Utilities/` isn't in the ShareExtension target. Filed PC-079 to deduplicate properly.

## Key finding for the squad — PC-080 is the unblocker
`clipTokenize()` in `Services/CLIPEmbeddingService.swift:283-308` is a fake byte-level mapping (`utf8_byte + 256`), not real CLIP BPE. The text encoder produces embeddings of byte-soup that the image encoder has never seen. Visual search has *never* worked correctly — every CLIP-dependent feature in the app silently returns garbage. Miguel confirmed: searching "dog" returned a Bach concert poster. PC-080 is the actual blocker for visual search shipping; everything else is paint over a hole.
